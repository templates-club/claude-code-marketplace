# Naming Conventions

Consistent naming makes skills easier to discover, reference, and maintain.

## Recommended: Gerund Form (Verb-ing)

The official recommendation is to use **gerund form** (verb + ing) because it clearly describes the activity or capability the skill provides.

### Good Examples

- `processing-pdfs` - clearly indicates PDF processing capability
- `analyzing-spreadsheets` - describes spreadsheet analysis activity
- `managing-databases` - shows database management function
- `testing-code` - indicates code testing activity
- `writing-documentation` - describes documentation writing

### Why Gerund Form?

- **Action-oriented**: Describes what the skill *does*
- **Clear intent**: Immediately understandable
- **Consistent**: Follows a predictable pattern
- **Professional**: Maintains a unified skill library

## Acceptable Alternatives

### Noun Phrases
- `pdf-processing`
- `spreadsheet-analysis`
- `database-management`

### Action Verbs
- `process-pdfs`
- `analyze-spreadsheets`
- `manage-databases`

## Avoid

### Vague Names
- ❌ `helper`
- ❌ `utils`
- ❌ `tools`
- ❌ `common`

### Overly Generic
- ❌ `documents`
- ❌ `data`
- ❌ `files`
- ❌ `processing`

### Reserved Words
- ❌ `anthropic-helper`
- ❌ `claude-tools`
- ❌ `anthropic-utils`

### Inconsistent Patterns
- ❌ Mixing `processing-pdfs` with `pdf-processor` in the same collection
- ❌ Using different naming styles across skills

## Format Rules

- **Lowercase only**: `processing-pdfs` not `Processing-PDFs`
- **Hyphens for separation**: `analyzing-data` not `analyzing_data` or `analyzingdata`
- **No spaces**: `testing-code` not `testing code`
- **Descriptive**: Be specific about what the skill does
- **Concise**: Keep names reasonably short (max 64 chars)
